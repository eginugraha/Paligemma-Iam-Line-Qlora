## 2. Pendahuluan & Latar Belakang
Penelitian HTR (*Handwritten Text Recognition*) tingkat baris (*line-level*) pada dataset internasional seperti **Teklia/IAM-line** memiliki tantangan tinggi berupa tingkat ambiguitas karakter yang disebabkan oleh variasi gaya tulisan tangan manusia (cakar ayam, miring, atau terputus). 

Untuk menyelesaikan masalah ini secara ilmiah, dibangun sebuah sistem berbasis *Vision-Language Model* **PaliGemma-3B-PT-448** yang dikombinasikan dengan teknik penalaran sekuensial *Chain of Thought* (CoT) dan sistem pengoreksi *Retrieval-Augmented Generation* (RAG). 

Aplikasi ini mengusung arsitektur terpisah (*decoupled architecture*) menggunakan **Svelte** di sisi *Front-End* untuk efisiensi memori, dan **FastAPI** di sisi *Back-End* untuk mengelola inferensi model GGUF lokal serta *database vector* **PostgreSQL (`pgvector`)** pada komputer lokal berspesifikasi RAM 8GB.

---

## 3. Alur Kerja Lingkungan (Hybrid Workflow)
Sistem ini menggunakan pembagian lingkungan kerja berbasis *hybrid* untuk menyiasati keterbatasan perangkat keras lokal:
1. **Fase Pelatihan (Cloud):** Proses komputasi berat dan *fine-tuning* model PaliGemma menggunakan metode **QLoRA** dilakukan di **Google Colab** memanfaatkan GPU NVIDIA T4. Output berupa *LoRA Adapter Weights* kemudian diunduh.
2. **Fase Implementasi (Lokal):** Proses inferensi dan UI dijalankan di **MacBook Pro Lokal (RAM 8GB)**. Model dasar PaliGemma dimuat dalam format **GGUF** untuk menghemat memori, digabungkan dengan adapter, dan diintegrasikan dengan database **PostgreSQL (`pgvector`)**.

---

## 4. Matriks Skenario Eksperimen (Objek Komparasi)
Sistem wajib memproses satu input gambar yang sama ke dalam **4 Kolom Skenario** secara paralel/sekuensial pada antarmuka pengguna:
* **M1 (Baseline QLoRA):** Inferensi visual langsung dari model PaliGemma yang telah di-*fine-tune*.
* **M2 (QLoRA + CoT):** Inferensi yang mewajibkan model menjabarkan karakteristik guratan huruf secara tekstual sebelum memberikan hasil akhir.
* **M3 (QLoRA + RAG):** Hasil teks dari M1 langsung dilempar ke PostgreSQL (`pgvector`) untuk dicari kosakata terdekat yang valid di kamus bahasa Inggris.
* **M4 (Hybrid - CoT + RAG):** Menggabungkan hasil penalaran huruf M2, lalu melakukan validasi kata akhir menggunakan pencarian vektor di PostgreSQL.

## 5. Kebutuhan Fungsional (Functional Requirements)

### 5.1. Modul Antarmuka Pengguna (Svelte Front-End)
* **FR-FE-01 (Upload Gambar):** Sistem harus menyediakan area *drag-and-drop* atau tombol unggah untuk file gambar tulisan tangan dengan format `.png`, `.jpg`, atau `.jpeg`.
* **FR-FE-02 (Loading State):** Sistem harus menampilkan indikator pemrosesan (animasi *spinner* atau *skeleton loading*) yang responsif selama FastAPI memproses data (maksimal 5 detik).
* **FR-FE-03 (Tampilan Komparasi 4 Kolom):** Sistem harus menampilkan hasil pemrosesan dalam bentuk 4 kolom berdampingan (*side-by-side*) setelah API memberikan respons JSON.
* **FR-FE-04 (Parameter Kolom):** Setiap kolom wajib menjajarkan **5 parameter komparasi** berikut secara vertikal:
    1. **Teks Hasil Akhir:** Output string alfabet final.
    2. **Nilai CER & WER Lokal:** Skor kesalahan karakter dan kata dari gambar tersebut yang dihitung langsung terhadap *ground truth*.
    3. **Waktu Pemrosesan (*Inference Time*):** Durasi eksekusi dalam satuan detik/milidetik.
    4. **Log Langkah Berpikir:** Teks penalaran CoT (untuk M2/M4) atau Log Query jarak kosinus PostgreSQL (untuk M3/M4).
    5. **Status Keberhasilan Koreksi:** Label indikator visual (`[Raw Output]`, `[Reasoned]`, `[Corrected]`, `[Optimal]`).
* **FR-FE-05 (Dashboard Statistik Global):** Sistem harus memiliki halaman/tab terpisah yang menyajikan tabel matriks ringkasan hasil pengujian massal (*batch evaluation*) untuk kebutuhan lampiran data Bab 4 Skripsi.

### 5.2. Modul API & Inferensi AI (FastAPI Back-End)
* **FR-BE-01 (Endpoint Inferensi - `POST /v1/detect`):** Endpoint yang menerima file gambar dari Svelte melalui skema `multipart/form-data`, mengeksekusi alur M1-M4, dan mengembalikan struktur data JSON terpadu.
* **FR-BE-02 (Integration Engine - PaliGemma GGUF):** Backend harus mampu memuat file base model PaliGemma 3B format GGUF dan menginjeksikan bobot file *LoRA Adapter* menggunakan *library* pengikat Python (`llama-cpp-python`).
* **FR-BE-03 (Metric Calculator):** Backend harus memiliki fungsi internal untuk menghitung *Character Error Rate* (CER) dan *Word Error Rate* (WER) secara *real-time* menggunakan algoritma *Levenshtein Distance* ketika data *ground truth* tersedia pada mode pengujian.

### 5.3. Modul Data & RAG (PostgreSQL Engine)
* **FR-DB-01 (Vector Storage):** Database harus menggunakan PostgreSQL dengan ekstensi `pgvector` yang aktif untuk menyimpan pustaka kata (*vocabulary*) bahasa Inggris dari dataset IAM-line.
* **FR-DB-02 (Vector Query):** Sistem harus mengeksekusi query pencarian jarak kedekatan vektor menggunakan operator jarak kosinus (`<=>`) untuk menemukan kecocokan string terdekat ketika fungsi RAG dipicu oleh FastAPI.
